## 1. Огляд проєкту

**LOGI-ANALYZER PRO** — це веб-додаток для аналізу збірних вантажних партій фармацевтичної та хімічної продукції, що транспортуються з Китаю/Індії в Україну транзитом через порти Євросоюзу (Гамбург, Роттердам, Гданськ). Додаток використовує AI (через зовнішні API) для аналізу маніфестів вантажів і автоматичного розрахунку митних платежів, оцінки ризиків, підготовки чекліста документів та перевірки відповідності нормам ЄС та України.

Весь додаток міститься в єдиному HTML-файлі (~4500 рядків) і не потребує бекенду — працює повністю на стороні клієнта (client-side).

## 2. Призначення та цільова аудиторія

**Цільова аудиторія:** логісти, митні брокери, спеціалісти ЗЕД (зовнішньоекономічної діяльності), імпортери фармацевтичної та хімічної продукції.

**Ключові задачі, які вирішує додаток:**

- Аналіз вмісту збірної партії з Excel/CSV файлів або Google Sheets
- Класифікація товарів за категоріями (АФІ, ветеринарна АФІ, косметична сировина, харчова добавка, допоміжна речовина, промислова хімія, подвійне призначення)
- Розрахунок митних платежів: митна вартість (CIF), мито, ПДВ 20%
- Оцінка ризиків для кожного товару (Критичний / Середній / Низький)
- Формування чекліста перевірок на митниці ЄС (фізичні та документальні)
- Формування чекліста розмитнення на митниці України
- Аналіз небезпечних вантажів (ADR-класифікація)
- Перевірка санкційних обмежень ЄС
- Генерація XLSX-звіту для завантаження
- Довідкова база знань з митного законодавства ЄС та України

## 6. Сторінки та навігація

Додаток має **3 сторінки** з навігацією через кнопки у верхній панелі:

### 6.1. Новий аналіз (`#page-new`)

Головна сторінка з полем вводу URL Google Sheets або завантаженням файлу. Містить:

- **API Key Card** — flip-картка для початкового вибору провайдера та введення ключа (зникає після збереження)
- **Command Deck** — центральна секція з заголовком, описом та формою вводу
- **Input Row** — поле URL + кнопка «Запустити аналіз»
- **File Upload** — альтернативне завантаження Excel/CSV файлу
- **Results Area** — область для динамічного відображення результатів

### 6.2. Архів (`#page-arch`)

Історія попередніх аналізів, збережена в localStorage (до 50 записів). Кожен запис відображає джерело, дату, кількість позицій, загальну суму та рівень ризику. Клік по запису відкриває збережені результати.

### 6.3. База знань (`#page-zed`)

Довідковий розділ із 8+ тематичними картками з інформацією про митне законодавство. Кожна картка відкривається у розгорнутому вигляді з розділами та парами ключ-значення.

## 7. Система API-ключів та провайдерів

### Підтримувані провайдери

| Провайдер | Модель | Формат ключа | Ендпойнт |
|-----------|--------|-------------|----------|
| OpenAI | `gpt-4o` | `sk-proj-...` | `api.openai.com/v1/chat/completions` |
| Gemini | `gemini-1.5-pro` | `AIzaSy...` | `generativelanguage.googleapis.com/v1beta/...` |
| Claude | `claude-opus-4-7` | `sk-ant-...` | `api.anthropic.com/v1/messages` |
| OpenRouter | `openai/gpt-4o` | `sk-or-...` | `openrouter.ai/api/v1/chat/completions` |

### Потік управління ключами

1. **Початковий стан** — видно flip-картку з вибором провайдера
2. **Вибір провайдера** — картка перевертається (CSS 3D-transform), з'являється поле вводу ключа
3. **Збереження** — ключ зберігається в localStorage, flip-картка ховається, з'являється API-чіп «ONLINE» у topbar
4. **Зміна ключа** — через модальне вікно Key Management (клік на API-чіп у topbar)
5. **Історія** — всі зміни ключів зберігаються у `keyHistory` (до 30 записів)

### Функція `callAI()`

Універсальна функція виклику AI, яка абстрагує різницю між API провайдерів:

- **OpenAI / OpenRouter** — стандартний Chat Completions API з `response_format: { type: 'json_object' }`
- **Gemini** — API `generateContent` з `responseMimeType: 'application/json'`
- **Claude** — Messages API із заголовком `anthropic-dangerous-direct-browser-access: true` (для прямого виклику з браузера)

Параметри: `systemMsg`, `userMsg`, `maxTokens`, `jsonMode` (true/false). Temperature завжди `0.0`.

## 8. Процес аналізу вантажу

Функція `runAudit()` координує повний процес аналізу:

```
1. Валідація (API ключ + джерело даних)
2. Показ лоадера (анімація маршруту EU → UA)
3. Отримання даних
   ├── Файл: використання вже завантажених даних
   └── Google Sheets: завантаження як XLSX через export URL
4. Парсинг SheetJS → масив листів [{name, rows}]
5. Пошук актуальних правил ЄС (AI-запит)
6. Розумна обробка листів:
   a. Парсинг дат з назв листів (DD.MM.YYYY)
   b. Пошук заголовків з "Номенклатура" + числові колонки
   c. Відбір табличних листів, сортування за датою (новіші першими)
7. Фільтрація «сміттєвих» рядків (isJunkRow)
8. Дедублікація товарів (normalizeItemName)
9. Батч-аналіз через AI (по 5 позицій за раз)
10. Пост-обробка: розрахунок CIF, мита, ПДВ
11. Рендеринг результатів
12. Збереження в архів
```

### Лоадер

Під час аналізу відображається анімований лоадер із:
- Візуалізацією маршруту EU → UA (два вузли з анімованою лінією між ними)
- Текстовими повідомленнями про поточний крок
- Підрядком з деталями (кількість листів, рядків, поточна партія)
- Спінером

## 9. Обробка даних

### 9.1. Завантаження Google Sheets

Для Google Sheets використовується URL експорту у форматі XLSX:

```
https://docs.google.com/spreadsheets/d/{sheetId}/export?format=xlsx
```

Таблиця повинна бути **публічною** (доступ «Всі з посиланням»). Після завантаження XLSX-буфер парситься через SheetJS, що дає всі листи з правильними назвами.

### 9.2. Завантаження файлів

Підтримуються формати `.xlsx`, `.xls`, `.csv`. Для Excel — SheetJS (завантажується динамічно з CDN при першому використанні). Для CSV — власний парсер `parseCSV()`.

### 9.3. Розумна обробка листів

**Парсинг дат із назв листів** — регулярний вираз шукає патерн DD.MM.YYYY в назвах листів. Листи сортуються за датою, щоб найсвіжіші дані мали пріоритет.

**Визначення табличних листів** — функція `findDataHeader()` шукає рядок-заголовок, що містить «Номенклатура» + хоча б одну з числових колонок (кол-ва, вес, ціна, кг, закупк, нетто).

**Фільтрація сміття** (`isJunkRow`) — відсіює рядки, які є коментарями, нотатками або заголовками, а не товарними позиціями. Перевіряє 30+ шаблонів (regex), серед яких: маркери заміток (`!!!`, `>>>`), логістичні коментарі («доставка», «всего 18т», «считаем»), повторні заголовки, рядки без числових даних у колонках кількості/ціни.

**Дедублікація** (`normalizeItemName`) — нормалізує назви товарів для виявлення дублікатів: видаляє статуси («заказано», «одобрено»), примітки в дужках, назви клієнтів/виробників, одиниці виміру, відсотки. Якщо той самий товар є в кількох листах — береться з найновішого.

## 10. AI-аналіз та системний промпт

### Системний промпт

AI отримує детальну інструкцію (системний промпт ~3000 слів) з такими ключовими блоками:

- **Роль:** професійний аналітик міжнародної логістики та митного оформлення
- **Контекст:** маршрут Китай/Індія → порти ЄС → Україна, дата — квітень 2026
- **Структура таблиць:** опис колонок (ЛС, Номенклатура, Кол-ва, Ціна, Виробник, МСДС/НЮАНСИ, код ТНВЭД)
- **Формат відповіді:** суворий JSON без markdown з полями `items[]`, `borderDelay`, `criticalAlert`, `nctsList`
- **Класифікація товарів:** 7 категорій із детальними критеріями
- **Ставки мита:** таблиця ставок за категоріями (АФІ 0–5%, хімія 5–6.5%, МКЦ 6.5% тощо)
- **Система ризиків:** Критичний (ADR, МСДС, подвійне призначення), Середній (GMP, CoA, температура), Низький (стандартні товари)
- **Перевірки ЄС та UA:** детальні приклади `euChecks` та `uaChecks` із світлофорною системою (green/yellow/red)
- **Актуальні правила ЄС:** результат попереднього AI-запиту підставляється в промпт

### Структура одного товару (JSON)

```json
{
  "name": "Назва товару з таблиці",
  "uctzedCode": "Код УКТЗЕД (10 цифр)",
  "qtyKg": 500,
  "buyPricePerKg": 12.50,
  "dutyRatePercent": 5.0,
  "category": "АФІ / Ветеринарна АФІ / ...",
  "applications": "Короткий опис застосування",
  "hazardAnalysis": "ADR клас або 'Безпечний. Не ADR.'",
  "storageRequirements": "Сухе місце, 15-25°C",
  "euChecks": [
    {"item": "Назва перевірки", "status": "green|yellow|red", "note": "Коментар"}
  ],
  "uaChecks": [
    {"item": "Назва перевірки", "status": "green|yellow|red", "note": "Коментар"}
  ],
  "risk": "Критичний|Середній|Низький",
  "riskNote": "Причина ризику"
}
```

### Батч-аналіз

Для великих маніфестів товари відправляються на AI **партіями по 5 позицій**. Функція `analyzeItemsInBatches()` розбиває дані, послідовно відправляє кожну партію і збирає результати. В останній партії AI також генерує загальні поля `borderDelay`, `criticalAlert`, `nctsList`.

### Пост-обробка (клієнтська)

Після отримання відповіді AI, система самостійно розраховує фінансові показники:

```
customsValue = qtyKg × buyPricePerKg × 1.10  (CIF = FOB × 1.10)
duty = customsValue × (dutyRatePercent / 100)
vat = (customsValue + duty) × 0.20
totalToPay = totalDuty + totalVAT
```

## 11. Рендеринг результатів

Результати відображаються у 3 секціях:

### 01 · OVERVIEW — Фінансова зведена

Чотири метричні картки:

| Метрика | Формула | Тег |
|---------|---------|-----|
| Митна вартість | Σ (FOB × 1.10) | CIF |
| Мито | Σ (CIF × ставка%) | УКТЗЕД |
| ПДВ | Σ (CIF + Мито) × 20% | 20% |
| До сплати | Мито + ПДВ | ИТОГО |

### 02 · MANIFEST — Склад та план перевірок

Таблиця з горизонтальним скролом та 6 колонками:

1. **Товар / УКТЗЕД** — назва, код, категорія, застосування
2. **К-сть · Ціна** — кілограми, ціна за кг, ADR-бейдж (якщо є)
3. **Вартість · Платежі** — митна вартість, мито (зі ставкою), ПДВ
4. **Перевірки в ЄС** — світлофорний список перевірок із легендою + info-картки (небезпека, зберігання)
5. **Розмитнення UA** — аналогічний світлофорний список для України
6. **Ризик** — пілюля-бейдж із кольором (червоний/жовтий/зелений) + причина

### 03 · ADVISORY — Ризики та чекліст

- **Критичний фактор** — червоний алерт (якщо є небезпечні товари)
- **NCTS Phase 5 · Checklist** — зелений алерт зі списком кроків

### Кнопка завантаження звіту

Кнопка з ripple-ефектом та анімацією стану (idle → downloading → success).

## 12. Експорт звіту в XLSX

| Аркуш | Зміст |

## 13. Архів перевірок

- Зберігається в `localStorage` під ключем `lap_archive`

## 19. Обмеження та відомі особливості

- **Немає бекенду** — API-ключі зберігаються в localStorage браузера (без шифрування). Це зручно, але потенційно небезпечно на спільних комп'ютерах.
- **CORS для Claude** — використовується нестандартний заголовок `anthropic-dangerous-direct-browser-access: true` для обходу CORS-обмежень Anthropic API при прямому виклику з браузера.
- **Google Sheets** — таблиця повинна бути публічною для доступу без автентифікації.
- **Батч-ліміт** — по 5 позицій на AI-запит, для великих маніфестів (100+ позицій) це може зайняти значний час.
- **Немає офлайн-режиму** — потребує інтернет-з'єднання для AI API та Google Fonts.
- **Немає мультимовності** — інтерфейс виключно українською мовою (з елементами англійської в технічних секціях).
- **Один файл** — увесь код в одному HTML-файлі (~4500 рядків), що ускладнює підтримку при подальшому розвитку.
